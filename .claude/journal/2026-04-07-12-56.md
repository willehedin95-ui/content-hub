## Decisions made
- **Source-of-truth via Meta API, not DB**: The new cron queries Meta directly across every active campaign rather than relying on `meta_campaigns.status='pushed'` like the old bleeder cron. This catches legacy ad sets, manually-created ad sets, and ad sets where the ad set is "off" but our DB still says they're live. The trade-off is API cost (lots of calls) — mitigated by throttling.
- **Protect permanent ad sets via `meta_campaign_mappings.is_permanent=true`**: Loaded at the start, never touched. This is the same protection mechanism used by the bleeder cron, so we have one source of truth for "do not pause."
- **Reuse `auto_paused_ads` table for logging**: Instead of a new table, log every pause to the same table the bleeder cron uses. The `meta_ad_id` column gets the ad set ID (slight schema overload, but acceptable for one shared audit log).
- **Throttle the read calls, not the write calls primarily**: The bulk of API consumption is `listAdsInAdSet()` (one per ad set). Even with only ~10 zombies pausing, the cron makes 180+ list calls in a typical run. Adding `await sleep(250)` after every list call adds ~45s of total runtime but eliminates the rate limit problem.
- **Did NOT add throttling to `listCampaigns`**: Only one call per workspace, no benefit.
